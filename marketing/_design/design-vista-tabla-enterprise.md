# vista-tabla-enterprise

## UX
VISTA TABLA ENTERPRISE (estilo Monday Main Table) para staff (admin/team). Sustituye renderList de app.js. Todo en espanol, sin em-dashes, edicion inline sin boton guardar, UI optimista con rollback y toast 'No se pudo guardar, intenta de nuevo.'

1) TOOLBAR DE VISTA (sticky bajo el topbar, dentro de #view): chip 'Agrupar: Mes' (opciones Mes | Estado | Sin grupos, abre picker), boton engrane 'Columnas' (lista de checkboxes de columnas visibles), menu de 3 puntos con 'Colapsar todo / Expandir todo', y contador 'N contenidos' (refleja busqueda activa del topbar). Todo persiste en localStorage por cliente.

2) GRUPOS COLAPSABLES: agrupacion default por Mes de publish_date (claves YYYY-MM ordenadas cronologicamente, grupo 'Sin fecha' SIEMPRE al final como backlog); por Estado usa STATUS_ORDER con color --st-*; posts con status desconocido caen en grupo 'Otros' (nunca invisibles, corrige el bug del kanban actual). Header de grupo: caret que rota, nombre ('Junio 2026' capitalizado es-MX o label del estado), color del grupo en caret + barra izquierda de 4px de todas sus filas, pill de conteo, y battery bar de distribucion de status (div flex con hijos flex-grow proporcionales, colores --st-*, title con desglose). Click en TODO el header colapsa/expande; colapsado deja visible header + battery (el board se lee de un vistazo). Estado de colapso persiste por cliente y por modo de agrupacion. Checkbox en el header selecciona todas las filas del grupo. Grupos vacios se omiten.

3) TABLA (escritorio): thead sticky; columna de checkbox + columna Titulo sticky left (background opaco) con boton 'Abrir' (drawer completo existente openEditorFor) y burbuja de comentarios; resto de columnas segun registro: Grabacion, Estado, Plataforma, Tipo, Fecha, Persona, Aprobacion, Caption, Notas <persona> (dinamicas desde note_labels), Inspo, Video. Render por registro de tipos: cada td lleva data-post-id y data-col; UN solo listener de click y UNO de keydown en el root (delegacion, no listeners por celda).

4) EDICION INLINE POR TIPO (cada commit = PATCH /posts/:id de UN campo, optimista, toast 'Guardado.' 1.2s, rollback en error; sin spinners):
- Estado: celda chip de ancho completo tenida con color-mix del --st-*; click abre picker con las 8 labels como bloques de color de ancho completo; un click aplica, cierra y dispara refreshClientCounts.
- Fecha: picker con input type=date nativo + atajos 'Hoy', 'Manana', '+7 dias', 'Quitar fecha'. Indicador de vencido (fecha pasada Y status distinto de publicado): texto rojo + icono, nunca solo color.
- Persona: picker con lista del staff (GET /users lazy, cacheado en store, avatares de iniciales) + input de texto libre como fallback (assignee sigue siendo TEXT libre, compatible con datos existentes); click asigna y cierra.
- Plataforma / Tipo / Grabacion / Aprobacion: pickers de opciones desde los mapas ENUM de api.js (PLATFORMS, CONTENT_TYPE_ORDER, Sin/G1-G5, APPROVALS; el staff puede forzar approval_state como hoy en el drawer).
- Titulo / Caption / Notas por persona: panel expandible con textarea autosize, Cmd/Ctrl+Enter guarda, Esc cancela, blur guarda; si el PATCH falla el panel queda abierto para no perder texto (contrato actual de openExpandPopover, portado a sheet.js).
- Inspo/Video: lectura como link target=_blank rel=noopener; click en zona vacia abre editor de URL.

5) SORT POR COLUMNA: click en th cicla asc, desc, ninguno (orden default position ASC + created_at). aria-sort + flecha visual. Sort es client-side y se aplica DENTRO de cada grupo (los grupos conservan su orden propio). Persiste en localStorage por cliente.

6) SELECCION MULTIPLE + ACCIONES MASIVAS: checkbox visible en hover (siempre visible cuando hay seleccion activa); shift+click selecciona rango (ancla guardada); checkbox de grupo; Esc limpia. Con N>0 aparece barra flotante inferior (pill centrada en desktop): 'N seleccionados' + acciones Estado, Fecha (submenu: 'Mover a fecha' con date input, o 'Mover N dias' con stepper +/-), Persona, Eliminar, y X para limpiar. Cada accion abre el mismo picker tipado y aplica OPTIMISTA a todos los seleccionados (snapshot completo previo; rollback total + toast si el server falla). Eliminar abre confirm modal con el conteo ('Eliminar 4 contenidos? Esta accion no se puede deshacer.') y llama bulk-delete. Backend valida pertenencia al cliente server-side; nunca N fetches.

7) FILA DE AGREGAR RAPIDO: ultima fila de cada grupo '+ Agregar contenido': se transforma en input inline; Enter hace POST /posts con defaults heredados del grupo (grupo mes: publish_date = dia 1 de ese mes; grupo estado: ese status; Sin fecha: publish_date null; tipo reel, position max del grupo + 1), inserta la fila optimista y deja el input activo para encadenar; Esc o blur con texto vacio cierra.

8) TECLADO (desktop): flechas mueven el foco entre celdas (roving tabindex con indices data-row/data-col), Enter abre el editor de la celda, Esc cierra editor o limpia seleccion, Tab/Shift+Tab avanzan. 

9) ESTADOS: skeleton durante loading (patron .skeleton existente); empty global con CTA 'Nuevo contenido'; busqueda sin resultados con empty propio; antes de cualquier re-render del cuerpo se cierran todos los pickers abiertos (sheet.closeAll(), evita los popovers huerfanos del codigo actual).

## Movil (390px)
DISENO PRIMERO A 390px (breakpoint pivote existente: max-width 720px). REGLA: la tabla NO se encoge; se transforma en lista de tarjetas-fila por grupo (patron Monday movil + auditoria del portal).

1) LAYOUT: bajo los .view-controls existentes (segmented + busqueda) va una fila de chips de control horizontal scrolleable: 'Agrupar: Mes', 'Orden: Fecha', 'Campos'. Cada chip abre un BOTTOM SHEET (componente sheet.js: translateY + transition 250ms, handle de arrastre, max-height 85dvh, overscroll-behavior contain, backdrop-dismiss solo en menus, nunca en formularios con texto).

2) GRUPOS: headers sticky (position sticky bajo los view-controls) con tap en TODO el header para colapsar (target completo, no solo la flecha), color del grupo, conteo y battery bar de status en CSS puro. Por defecto colapsados EXCEPTO el grupo del mes actual (o el primero con contenido): la pantalla inicial es un indice con baterias, se lee el estado de todo el mes de un vistazo. Colapso persistido en localStorage por cliente.

3) TARJETA-FILA (min-height 64px, targets 44px): barra izquierda 4px del color del grupo; linea 1 titulo truncado; linea 2: chip de Estado (tap DIRECTO abre bottom sheet con las 8 labels como filas de color de ancho completo, 48px cada una, un tap aplica y cierra), fecha corta tocable (tap abre sheet con input type=date nativo de iOS/Android + atajos), avatar de iniciales del assignee, approval badge. Estado y fecha se editan con tap directo porque son el 80% de los toques; tap en CUALQUIER otra zona de la tarjeta abre el editor drawer existente (bottom sheet 92dvh ya implementado en app.css). Sin edicion de texto inline con teclado en lista: titulo y textos largos se editan en el drawer.

4) SELECCION MASIVA: long-press ~500ms sobre una tarjeta (timer en touchstart, cancelado si el dedo se mueve mas de 10px = scroll; navigator.vibrate(10) al activar) entra en modo seleccion: aparecen checkboxes en todas las tarjetas y tap = toggle. Fallback visible obligatorio (gesto oculto nunca es el unico camino): opcion 'Seleccionar' en el menu de 3 puntos del header de grupo. Barra de acciones fija abajo con padding-bottom env(safe-area-inset-bottom): 'N' + iconos Estado, Fecha, Persona, Eliminar, X; cada accion abre bottom sheet; 'Mover N dias' con stepper grande +/-. Tres listeners touch delegados en el contenedor de la lista, no por tarjeta.

5) AGREGAR RAPIDO: boton '+ Agregar' al final de cada grupo expandido (siempre visible, hover no existe en touch) abre bottom sheet de quick-create: input de titulo con autofocus + chips de defaults editables (fecha del grupo, tipo); 'Crear' o Enter crea y deja el sheet listo para encadenar otro. El teclado no tapa el boton (boton sticky bottom dentro del sheet).

6) CAMPOS EN TARJETA: el sheet 'Campos' controla que 2 chips secundarios muestra la tarjeta (default Estado + Fecha; alternativas Plataforma, Tipo, Grabacion, Aprobacion). Maximo 3 campos visibles a 390px, con ellipsis.

7) SORT/ORDEN: sheet con opciones radio (Fecha, Estado, Titulo, Grabacion, Posicion) + direccion. El indicador del orden activo vive en el chip ('Orden: Fecha').

8) DETALLES: scroll vertical unico (touch-action pan-y); sin drag and drop en v1 movil (mover fecha/estado = picker, mas fiable que drag en 390px, decision alineada con Monday movil); usar 100dvh, prefers-reduced-motion respetado; toasts encima de la barra de seleccion; vencidos con texto rojo + icono (AA, no solo color).

## Contrato de datos
LEE del store compartido (js/store.js, integrador): state.me {id, role} (gate: solo admin/team llegan a esta vista); state.activeClientId; state.clients (activeClient().note_labels para columnas de notas dinamicas, brand_color para el color de grupos por mes); state.posts (TODOS los posts del cliente activo, shape completo de mkt_posts: id, title, content_type, grabacion, publish_date 'YYYY-MM-DD', assignee, platform, status, caption, inspo_url, video_url, hook, body, cta, hashtags, notes_team, notes_people {}, client_visible, approval_state, position, created_at); state.search (filtro client-side title||caption); state.users (cache lazy: la vista pide store.loadUsers() -> GET /users la primera vez que se abre el picker de persona); state.loading.

ESCRIBE via acciones del store (todas optimistas con snapshot + rollback + toast, emiten 'posts:changed' y disparan refreshClientCounts best-effort):
- store.patchPost(id, {campo: valor}) -> PATCH /api/marketing/posts/:id (un campo por commit; notes_people se parchea como objeto completo tras merge inmutable, patron actual).
- store.createPost({client_id, title, content_type:'reel', status, publish_date|null, position}) -> POST /api/marketing/posts (normaliza created.post||created, defaults completos).
- store.bulkUpdate(ids[], patch) -> POST /api/marketing/posts/bulk-update (NUEVO; patch whitelisted: status, publish_date, assignee, platform, content_type, grabacion, client_visible).
- store.shiftDates(ids[], dias) -> calcula publish_date nuevo por post en cliente y manda POST /api/marketing/posts/bulk-update {updates:[{id, publish_date}]} (forma por-item del mismo endpoint).
- store.bulkDelete(ids[]) -> POST /api/marketing/posts/bulk-delete (NUEVO).

LOCAL (localStorage, por dispositivo, claves namespaced): 'mkt.table.groupBy.<clientId>' ('month'|'status'|'none', default 'month'); 'mkt.table.collapsed.<clientId>.<groupBy>' (array de claves de grupo colapsadas); 'mkt.table.sort.<clientId>' ({key, dir}|null); 'mkt.table.cols.<clientId>' (array de keys de columnas visibles, desktop) y 'mkt.table.cardFields.<clientId>' (max 2 chips de tarjeta movil). EFIMERO (memoria del modulo, NUNCA en store): Set de ids seleccionados, ancla de shift-click, modo seleccion movil, picker abierto.

EVENTOS consumidos: 'posts:changed' (re-render del cuerpo preservando seleccion valida y colapsos; cierra pickers via sheet.closeAll() antes de replaceChildren); 'client:changed' (resetea seleccion y relee localStorage del cliente nuevo). La vista expone una unica funcion export: renderTableView(rootEl) registrada en el router de vistas del shell.

## API
- POST /api/marketing/posts/bulk-update (NUEVO): body {ids:[...], patch:{...}} o {updates:[{id, publish_date}]}; gate isStaff; valida con SELECT previo que todos los ids existan y pertenezcan a UN solo client_id (devuelve 400 con ids invalidos, mejora sobre el no-op silencioso de /posts/reorder); whitelist de campos = subset de POST_EDITABLE_FIELDS (status, publish_date, assignee, platform, content_type, grabacion, client_visible) validado contra enums; aplica con env.DB.batch() atomico + updated_at; UNA fila logActivity 'post.bulk_update' con conteo y detalle; respuesta {ok, updated, posts:[shapePost]}. Rutearlo en el bloque parts[0]==='posts' && parts.length===2 ANTES del matcher /posts/:id, junto a 'reorder'.
- POST /api/marketing/posts/bulk-delete (NUEVO): body {ids:[...]}; gate isStaff; misma validacion de existencia/pertenencia; batch DELETE (FK cascade limpia comments/approvals); logActivity 'post.bulk_delete' con conteo; respuesta {ok, deleted}.
- REUSADOS sin cambios: GET /posts?client_id, PATCH /posts/:id (parcheo por campo), POST /posts (quick add), GET /users (picker de persona), GET /clients (note_labels, counts), POST /posts/reorder (queda como esta; el calendario lo sigue usando).
- SIN migracion D1 en este modulo: agrupacion, sort, columnas y colapso persisten en localStorage v1; las vistas guardadas en D1 (mkt_saved_views) quedan para el modulo de vistas en 004+ y este modulo las adoptara leyendo el mismo shape {group_by, sort, cols}.
