# shell-nav-design-system-v2

## UX
PRINCIPIO: el shell es el host de toda la app enterprise. Provee router, store, prefs, topbar, bottom-nav, sheets, toasts y tokens; las vistas (calendario, tablero, tabla, dashboard, mi-trabajo) son plugins que se registran via shell.registerView(). Cero frameworks, ES modules, todo dato de usuario via textContent (patron el() de api.js, innerHTML solo para SVG estatico en icons.js).

1) BOOT (shell.js): app.html muestra #boot (splash actual). main.js importa shell y las vistas registradas. boot(): GET /auth/me; 401 o red caida -> location.replace('/marketing/'); role client -> location.replace('/marketing/client'); admin/team pasan. En paralelo: GET /clients y GET /notifications/unread-count (best-effort, si 404 el badge se oculta: permite desplegar shell antes que la migracion 004). prefs.migrate() lee las claves viejas mkt.view y mkt.client una sola vez y las copia al namespace nuevo mkt2.<userId>.*. Router resuelve el hash inicial (deep link gana sobre prefs; sin hash usa prefs lastClient + lastView.<clientId>, default tablero). Render chrome (topbar + bottomnav), mount de la vista activa, fade-out de #boot.

2) ROUTER POR HASH (router.js): formato #/<vista>?cliente=<id>&q=...&estado=...&desde=...&hasta=... mas ruta especial #/post/<id> (detalle full-screen, la monta el modulo editor pero el shell la rutea). parse() -> {view, params}; navigate(view, params, {replace}) serializa y asigna location.hash; listener hashchange -> diff: si cambio view desmonta vista anterior (unmount) y monta nueva, si solo cambiaron params llama view.onParams(params) sin remount. Los filtros viven en la URL (compartible/bookmarkable); el store los espeja en state.filters. CAPAS Y BOTON ATRAS: cada sheet/overlay abierto hace history.pushState({layer:n}); popstate con layer pendiente cierra la capa superior (sheet, busqueda, panel de avisos) y NO navega; asi el boton atras del telefono siempre hace lo esperado. Vista desconocida en hash -> redirect silencioso a tablero + toast info.

3) TOPBAR (topbar.js, 56px fija arriba, blur, fondo rgba bg .85): IZQUIERDA: switcher de cliente = pill 44px con dot 10px del brand_color (halo via color-mix), nombre truncado y chevron. Tap -> sheet 'Tus clientes' (clientswitcher.js): si hay mas de 8 clientes, input de filtro arriba; filas 56px con dot de marca, nombre, badge gradiente con counts.pending si > 0; primera opcion 'Todos los clientes' (solo staff, habilita vistas globales: cliente=todos en URL); footer '+ Nuevo cliente' (abre el modal de cliente existente). Seleccionar: cierra sheet, set activeClientId optimista, persiste pref lastClient, actualiza ?cliente= con replace, emite evento client:changed (las vistas re-fetchan), resetea q. El borde inferior del topbar lleva un tinte de 2px del brand color activo (color-mix 35% transparent): la marca del cliente es tinte, el gradiente violeta-rosa IVAE sigue siendo el unico color de accion. DERECHA: boton lupa 44px, campana 44px con badge rojo (numero, '9+' si excede, aria-label con conteo), avatar 32px. Tap avatar -> sheet de cuenta: nombre + rol, filas Equipo (modal existente), Actividad (modal existente), Ajustes de avisos (placeholder que abre tab de prefs del panel de notificaciones), Cambiar contraseña, Salir (POST /auth/logout + replace /marketing/). DESKTOP >=1024px: ademas de lo anterior aparecen tabs de vista inline (Calendario Tablero Tabla Dashboard) con subrayado gradiente en la activa y la busqueda como input inline de 280px con dropdown de resultados anclado.

4) SWITCHER DE VISTAS DE CONTENIDO: en movil las vistas calendario/tablero/tabla comparten el tab 'Contenido' del bottom-nav; el shell renderiza un sub-header sticky (debajo del topbar, z 15) con segmented control de 3 opciones, targets 44px, que llama navigate(view) y persiste pref lastView.<clientId>. Este sub-header es un slot del shell (shell.setViewControls(nodes)) para que cada vista pueda añadir su boton de filtros con badge sin duplicar logica de foco (leccion del codigo actual: los controles moviles re-renderizan solo el body de la vista, nunca el sub-header, para no perder el foco del input).

5) BUSQUEDA GLOBAL (search.js): tap en lupa -> overlay full-screen (movil) con input autofocus, boton Cancelar, y debajo 'Busquedas recientes' (prefs recentSearches, max 8, tap re-ejecuta, X borra). Debounce 250ms. Fase 1 instantanea: filtra en memoria los posts del cliente activo (title, caption, hook) y pinta seccion 'En <cliente>' con filas: titulo, chip de tipo, badge de estado, fecha corta. Fase 2 (solo staff, query >= 2 chars): GET /search?q= -> secciones 'Otros clientes' (posts con dot del cliente) y 'Clientes' (nombre + handle). Tap en post -> navigate('#/post/<id>') con ?cliente= correcto; tap en cliente -> cambia cliente activo y va a tablero. Sin resultados: empty amistoso 'No encontramos nada con esa palabra'. Esc, Cancelar o atras cierran (capa history). En desktop el mismo modulo renderiza dropdown anclado en vez de overlay.

6) CAMPANA / AVISOS (notifications.js): badge alimentado por GET /notifications/unread-count cada 60s con setTimeout encadenado (no setInterval), refetch inmediato en visibilitychange visible y tras cada mutacion propia (hook store.emit('mutated')). El conteo se pinta en campana, en el tab Avisos del bottom-nav y como prefijo '(N) ' en document.title. Tap campana o tab Avisos -> panel: en movil overlay full-screen con header sticky (titulo Avisos + boton 'Marcar todas como leidas'), segmented control Todas / No leidas / Menciones / Asignadas (mapea a ?filter=). Filas 56px: avatar con iniciales del actor, cuerpo resuelto server-side en 2 lineas max ('Vianey aprobo Reel Cenotes'), dot del cliente, tiempo relativo, punto azul + fondo tintado si no leida. Tap en fila: POST /notifications/read {ids:[id]} optimista (dot desaparece al instante, rollback si falla) y deep-link al post (#/post/<id>, anclado a comentario si comment_id). Boton 3 puntos 44px por fila -> action sheet: marcar leida/no leida, eliminar. Tab 'No leidas': una fila recien leida permanece visible atenuada hasta cerrar el panel (patron Monday, permite revertir). Vacio: 'Estas al dia' con icono. Desktop: popover anclado de 380px con el mismo contenido. Si el endpoint devuelve 404 (migracion 004 no aplicada) el shell oculta campana y tab Avisos pasa a 3 tabs: degradacion limpia.

7) BOTTOM NAV (bottomnav.js): 4 tabs: Inicio (dashboard), Contenido (ultima vista de contenido usada), Mi trabajo, Avisos. fixed bottom, height 56px + padding-bottom env(safe-area-inset-bottom), fondo blur, iconos 24px + label 10px, target completo 44px+, aria-current=page. Badges: Avisos = no leidas (rojo), Mi trabajo = vencidos (lo provee el modulo mi-trabajo via shell.setTabBadge('mi-trabajo', n)). Re-tap en tab activo: scroll suave al tope del contenedor de la vista. FAB: shell expone shell.setFab({label, onTap}) y lo pinta como boton gradiente 56px fixed right 16px bottom calc(nav + 16px + safe-area); cada vista decide su accion (quick add); si la vista no setea FAB no se muestra. El FAB y los toasts nunca se solapan (toast se posiciona encima).

8) SISTEMA DE SHEETS (sheet.js, EL componente mas reutilizado): API unica openSheet({title, build(bodyEl, close), mode:'menu'|'form'|'picker', onClose}) -> {close, el}. MOVIL: panel fixed bottom, border-radius 18px arriba, asa 40x4px, max-height 85dvh, entrada translateY(100%)->0 en 250ms ease-out, backdrop rgba .6; area scrolleable con overscroll-behavior contain; drag del asa con pointer events (cerrar si arrastra > 30% o velocidad alta). REGLAS NN/g: mode menu/picker cierra con tap en backdrop y con atras; mode form NO cierra por backdrop (solo X o Cancelar) para no perder datos. Focus trap, Esc cierra la capa superior, scroll lock del body, devuelve foco al disparador. DESKTOP >=768px: mode menu/picker se renderiza como popover anclado (reusa la logica flip/clamp de openPopover actual) y mode form como modal centrado. Helper de alto nivel pickFrom({title, options:[{value,label,color,icon,current}]}) -> Promise<value|null>: es el contrato que usaran status pickers, fecha, persona y acciones masivas de los otros modulos. Solo puede existir 1 sheet + 1 picker apilado; abrir un tercero reemplaza al segundo.

9) TOASTS CON DESHACER (toast.js): toast(msg, {type:'success'|'error'|'info', ms=4000, action:{label:'Deshacer', onAction}}). Host aria-live polite, posicion bottom calc(56px + safe-area + 12px) centrado, max 2 visibles en cola. Con action: duracion 5000ms, boton de texto en gradiente, tap ejecuta onAction (tipicamente el rollback devuelto por store.optimistic), cierra el toast y muestra 'Deshecho' 1.5s. Tap en el cuerpo descarta. Patron oficial de mutacion para TODOS los modulos: const rollback = store.optimistic(apply); api.patch(...).catch(() => { rollback(); toast('No se pudo guardar, intenta de nuevo', {type:'error'}) }); en exito toast con Deshacer cuando la accion es reversible (mover fecha, cambiar estado, archivar).

10) MANEJO GLOBAL DE SESION Y RED: el shell instala un interceptor (wrapper fino sobre api.*): err.status 401 fuera de boot -> guarda hash actual en prefs returnTo, toast 'Tu sesion expiro' y replace a /marketing/ (arregla el gap actual de toasts infinitos); login futuro puede leer returnTo. Eventos online/offline: barra fina ambar 'Sin conexion' bajo el topbar, se oculta al volver; en offline los botones de mutacion no se deshabilitan (optimista igual encola error y rollback).

11) DESIGN TOKENS V2 (tokens2.css, capa aditiva sobre :root de app.css, NO duplica los tokens v1): identidad intacta (violet/fuchsia/pink, fondos dark). Nuevos: --nav-h:56px, --topbar-h2:56px, --subhead-h:48px; escala tipografica --fs-1..--fs-6 (12/13/15/17/20/24); elevaciones --e1..--e4; --sheet-radius:18px; --tap:44px; --client-accent (la inyecta shell.js desde brand_color validado hex 3/6/8, fallback --brand) con derivados --client-accent-soft via color-mix; z-index nuevos documentados en los huecos de la escala existente: sub-header 15, bottom-nav 90, sheet-backdrop 138, sheet 140, search overlay 150, notif overlay 160 (debajo de toast 200 y boot 300); [data-density='compact'] reduce paddings y alturas de fila (pref de usuario, hook para la tabla); [data-theme='light'] declarado vacio como punto de extension, no se embarca. Todo color nuevo via var + color-mix, sin hex sueltos. UI 100% en espanol, sin em-dashes en ningun string visible.

12) ACCESIBILIDAD: focus-visible con --ring existente, roles correctos (tablist en segmented, dialog en sheets, status en toasts), prefers-reduced-motion respeta el bloque global existente, contraste AA en badges y labels del bottom-nav, targets 44px universales.

## Movil (390px)
Disenado primero a 390px, desktop es la adaptacion. LAYOUT BASE: body 100dvh (nunca 100vh), un solo contenedor de scroll #viewScroll con padding-top var(--topbar-h2) (+ --subhead-h cuando la vista de contenido muestra segmented) y padding-bottom calc(var(--nav-h) + env(safe-area-inset-bottom) + 16px). Desaparece el sidebar drawer actual: en movil la navegacion top-level vive en el bottom-nav (zona del pulgar, 49% de usuarios usan un pulgar) y los clientes en el switcher del topbar. TOPBAR 390px: una sola fila 56px: [switcher cliente flexible truncado] [lupa 44] [campana 44 + badge] [avatar 32]; sin tabs ni input de busqueda inline. SUB-HEADER de vistas de contenido: sticky top 56px, segmented Calendario/Tablero/Tabla con botones flex 1 de 44px + slot para boton Filtros con badge numerico; full-bleed con blur (patron .view-controls actual conservado). BOTTOM-NAV: 4 tabs de 25% cada uno, icono 24 + label 10px, badge rojo arriba-derecha del icono; safe-area inferior; siempre visible (no se oculta al scrollear: la orientacion vale mas que 56px). FAB 56px gradiente, right 16px, bottom calc(56px + 16px + safe-area); accion la define la vista activa. SHEETS: max-height 85dvh, asa arrastrable, animacion 250ms, los pickers usan filas de 48px minimo con el color a ancho completo (imposible fallar el tap); teclado abierto no tapa botones de submit (footer del sheet position sticky bottom). BUSQUEDA: overlay full-screen, input 48px autofocus con teclado abierto de inmediato, Cancelar visible, resultados como filas 56px; cierre con atras del telefono (capa history). AVISOS: overlay full-screen, segmented deslizable horizontal si no caben 4 tabs, filas 56px tap completo, acciones secundarias tras boton 3 puntos 44px (hover no existe en touch, nada depende de hover en todo el shell). TOASTS: encima del bottom-nav, nunca tapados por teclado (al abrir teclado se reposicionan con visualViewport si esta disponible, fallback bottom fijo). GESTOS: swipe-down cierra sheets (pointer events, umbral 30%); sin pull-to-refresh custom (refetch en visibilitychange + el PTR nativo del browser sigue vivo); sin long-press en el shell (reservado a las vistas). RENDIMIENTO: chrome del shell se monta 1 vez y nunca se re-renderiza completo (solo parches quirurgicos: badge, nombre de cliente, tab activa); listeners delegados en topbar/bottomnav/sheet host (6 listeners totales del shell); transiciones solo transform/opacity. BREAKPOINTS (mobile-first, min-width): base <768 todo lo anterior; >=768 sheets de pickers pasan a popover anclado, bottom-nav persiste; >=1024 bottom-nav se oculta, aparecen tabs de vista + busqueda inline en topbar y el switcher de cliente como popover; el FAB se vuelve boton 'Nuevo contenido' en el topbar. iOS Safari: viewport-fit=cover ya esta en app.html, safe-areas en nav/FAB/toast, scroll lock de sheets via overflow hidden en body + overscroll-behavior contain (evita rubber-band del fondo).

## Contrato de datos
STORE (js/shell/store.js, singleton exportado): state = { me:{id,email,name,role,client_id}, clients:[shapeClient con counts.{posts,pending}], activeClientId:string|'todos', view:string, params:{}, unreadCount:number, online:bool, booted:bool }. API: getState(); set(patch,{silent}); subscribe(keys[]|'*', fn)->unsub (notifica con (key, value, prev)); optimistic(applyFn)->rollbackFn (captura snapshot superficial de las claves tocadas antes de aplicar); emit(event,payload)/on(event,fn) bus para eventos de dominio: 'client:changed', 'posts:changed' (lo emiten las vistas tras mutar, el shell refresca counts con GET /clients best-effort, sustituible luego por /stats), 'mutated' (dispara refetch de unread-count), 'notifications:read'. Las vistas NUNCA tocan localStorage ni location.hash directo: todo via prefs y router.

PREFS (js/shell/prefs.js): namespace mkt2.<userId>.<clave> en localStorage, JSON seguro (try/catch, default). Claves del shell: lastClient, lastView.<clientId>, recentSearches (array max 8), density ('comoda'|'compacta'), returnTo. Claves reservadas para vistas (mismo helper): collapsedGroups.<view>.<clientId>, boardCols.<clientId>, calMode.<clientId>. migrate(): copia una vez mkt.view -> lastView y mkt.client -> lastClient y no borra las viejas (rollback posible).

ROUTER (js/shell/router.js): hash #/<vista>?cliente=&q=&estado=&tipo=&persona=&desde=&hasta= y #/post/<id>. registerView({id, label, icon, badgeKey?, mount(el,ctx), unmount?(), onParams?(params)}) lo llama main.js por cada modulo de vista. ctx entregado a mount = { store, prefs, router:{navigate,current}, sheet:{openSheet,pickFrom}, toast, setFab, setViewControls, setTabBadge, icons }. El router fuerza coherencia: si params.cliente difiere de activeClientId actualiza el store primero y luego monta.

LECTURAS API DEL SHELL: GET /auth/me (gate); GET /clients (switcher, badges pending, brand_color; contrato existente intacto); GET /notifications?filter=all|unread|mentions|assigned&limit=50 -> [{id,type,post_id,comment_id,client_id,actor_name,body,read,created_at}] (body YA resuelto server-side, render con textContent); GET /notifications/unread-count -> {count}; GET /search?q= -> {posts:[{id,title,client_id,client_name,status,content_type,publish_date}], clients:[{id,name,instagram_handle,brand_color}]} (solo staff; rol client jamas llega al shell). ESCRITURAS API DEL SHELL: POST /notifications/read {ids:[..]} o {all:true}; POST /auth/logout. Todo lo demas (posts, comments, reorder) pertenece a los modulos de vista, que usan api.js directo + el patron optimistic/rollback/toast del shell.

CSS: tokens2.css solo agrega variables y atributos de tema/densidad; shell.css consume tokens v1 + v2; --client-accent la escribe shell.js en document.body.style tras validar hex (misma validacion del portal cliente). COMPAT: api.js se importa sin cambios (BASE /api/marketing, cookie mkt_session same-origin, mapas ENUM como fuente unica); client.html/client.js NO se tocan; el aislamiento de cliente sigue 100% server-side (el shell nunca envia client_id para rol client porque ese rol nunca entra a app).

## API
- NUEVO GET /api/marketing/notifications?filter=all|unread|mentions|assigned&limit=50: scoped SIEMPRE por session.user_id, devuelve filas de mkt_notifications (migracion 004) con body ya resuelto; indice (user_id, read, created_at)
- NUEVO GET /api/marketing/notifications/unread-count -> {count}: query barata indexada, es la que se pollea cada 60s
- NUEVO POST /api/marketing/notifications/read {ids:[..]} | {all:true}: marca read=1 solo de filas del propio user_id
- NUEVO GET /api/marketing/search?q=: solo isStaff (rol client recibe 403, nunca lo llama el shell), LIKE sobre mkt_posts.title/caption + mkt_clients.name/instagram_handle, limit 20, devuelve {posts, clients} con client_name desnormalizado
- REUSADOS sin cambios: GET /auth/me, GET /clients (con counts.pending y brand_color), POST /auth/logout, POST /auth/change-password
- Las rutas nuevas se agregan en route() de functions/api/marketing/[[path]].js como rama parts[0]==='notifications' y parts[0]==='search' antes del 404 final, gateadas con getSession; la tabla mkt_notifications nace en la migracion 004 del modulo de automatizaciones/notificaciones (este modulo solo la consume y degrada si falta)
