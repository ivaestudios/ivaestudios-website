# dashboard

## UX
VISTA NUEVA "Resumen" dentro del app de equipo (app.html), registrada como cuarta pestana del segmented control (calendar/board/list/dashboard) y propuesta como vista por defecto al hacer boot para admin/team (whitelist en boot() de app.js). El portal de cliente NO la usa en v1 (endpoint staff-only); queda preparada para una variante reducida v2.

ESTRUCTURA Y FLUJO: al entrar, mountDashboard(container, ctx) pinta skeleton cards (reusa .skeleton de app.css) y hace UN solo fetch a GET /api/marketing/dashboard. Header propio del dashboard, sticky bajo los view-controls: (a) toggle de alcance como segmented de 2 botones [<Cliente activo>] [Agencia], visible solo para staff con mas de 1 cliente; persiste en localStorage 'mkt.dash.scope'; en alcance cliente usa state.activeClientId; (b) navegador de mes: chevron anterior, label 'Junio 2026' (es-MX capitalizado), chevron siguiente, boton 'Este mes' que regresa al mes actual; persiste 'mkt.dash.month' solo en memoria (cada entrada a la vista abre en el mes actual, decision deliberada: el dashboard responde 'como vamos HOY'). Cambiar scope o mes = un re-fetch del payload completo y re-render de todos los widgets (replaceChildren del body del dashboard, nunca del header, para no perder foco/scroll del toggle).

Orden: clientes con atrasados primero, luego por pendientes desc, luego alfabetico. Tap en la card = ctx.selectClient(id) + queda en dashboard con scope cliente (drill-down natural). Despues de las cards: donut de plataforma global y racha global. Clientes archivados excluidos server-side.

ESTADOS TRANSVERSALES: carga = skeletons por widget; error de fetch = card unica con mensaje en espanol y boton 'Reintentar'; mes sin ningun post = empty global con CTA crear. Re-fetch automatico en visibilitychange (document visible) y boton refresh discreto en el header. Sin polling. Cero spinners bloqueantes: el payload llega completo en una respuesta (sin waterfall por widget).

Una sola columna, padding 16px, todo apilado en el orden de importancia descrito (contadores, pipeline, por aprobar, semana, atrasados, donut, racha; en global: contadores, cards de cliente, donut, racha). CONTADORES en grid 2x2 (no apilados de a uno como Monday movil: desperdicia pantalla); cada card minimo 76px de alto, area de tap completa. HEADER del dashboard sticky (top: 0 dentro de .main__scroll, bajo los view-controls existentes que ya son sticky z-10; el header del dash usa z-9 y fondo rgba(11,11,18,.92) + blur, mismo patron view-controls): toggle de scope con botones de 44px y navegador de mes con chevrons de 44x44. SIN tooltips de hover en ningun widget: bateria y donut llevan leyenda textual siempre visible debajo (label + conteo + %), cumpliendo no-solo-color; los segmentos de la bateria ademas son tocables con target ampliado via padding vertical (la barra mide 14px pero el hit area 44px). Listas (por aprobar, semana, atrasados) con filas de minimo 56px, titulo con ellipsis a 1 linea, chips compactos; tap en TODA la fila navega (editor), la accion secundaria 'Reprogramar' de atrasados es boton propio de 44px a la derecha con stopPropagation. El popover de reprogramar se ancla con el patron gpop existente (flip/clamp al viewport) y usa input type=date nativo de iOS/Android (cero JS de calendario). Donut 160px centrado con leyenda apilada debajo (no a la derecha). Sparkline de racha: 14 barras de minimo 12px de ancho con gap 4px (cabe justo en 390 - 32 de padding - 32 de card). Cards de cliente (global) a ancho completo, bateria de 10px, badges como pills de 24px; con 6+ clientes la lista scrollea normal, sin virtualizacion (escala actual lo permite). Padding-bottom final: calc(24px + env(safe-area-inset-bottom)). DESKTOP (>=900px, enhancement segundo): grid de 12 columnas con gap 16: contadores 4x span-3; bateria span-8 + racha span-4; por aprobar span-6 + atrasados span-6; semana span-7 + donut span-5; cards de cliente en grid de 2 columnas. Breakpoint unico adicional 720px ya usado por el resto del app para detalles menores. prefers-reduced-motion respeta el global existente (sin animacion de entrada de barras).

## Contrato de datos
LEE DEL STORE/CONTEXTO (no muta nada global): mountDashboard(container, ctx) recibe ctx = { state (lee state.me.role, state.activeClientId, state.clients para nombre/color en header y fallback), setView(view), openEditorFor(id|null), openActivityModal(), selectClient(id) } provisto por app.js. Estado propio del modulo (no compartido): dashState = { scope: 'client'|'global', month: 'YYYY-MM', data: payload|null, loading, error }. Persistencia: localStorage 'mkt.dash.scope'; sessionStorage 'mkt.jump' (escribe JSON {view, filters:{approval?, status?, platform?, from?, to?}} para drill-down; las otras vistas lo consumen y borran una vez, contrato opcional y degradable).

API QUE CONSUME: (1) GET /api/marketing/dashboard?month=YYYY-MM[&client_id=ID] via api.get de api.js (cookie mkt_session, timeout 30s, errores en espanol). Sin client_id y rol staff = alcance agencia. Gate: isStaff en v1; un rol client recibe 403 (el portal tendra variante reducida en v2 con publicPost-allowlist). (2) PATCH /api/marketing/posts/:id {publish_date} (endpoint EXISTENTE, sin cambios) para 'Reprogramar' optimista.

SHAPE DE RESPUESTA (todo en UNA respuesta, calculado con env.DB.batch de 6 queries en alcance cliente y 8 en global, una sola ida a D1):
{
  scope: 'client'|'global', month: '2026-06', today: '2026-06-09' (fecha local America/Cancun calculada en el Worker, UTC-5 fijo sin DST),
  counters: { pending: n, week: n, overdue: n, monthTotal: n, noDate: n },
  pipeline: [{ status, count }] (incluye solo status con count>0; status fuera del enum se reportan tal cual y el frontend los agrupa en 'otros'),
  approvals: { count, items: [postLite x5] },
  week: { items: [postLite x20 max] },
  overdue: { count, items: [postLite x10, ordenados mas viejo primero] },
  platforms: [{ platform: 'Instagram'|'TikTok'|'Facebook'|'YouTube'|'LinkedIn'|'Otra', count }] (normalizado server-side con TRIM + match case-insensitive contra PLATFORMS),
  activity: { streak: n, days: [{ d: 'YYYY-MM-DD', count }] (ultimos 14 dias, huecos con 0) },
  clients: [{ id, name, brand_color, monthTotal, pending, overdue, nextDate: 'YYYY-MM-DD'|null, pipeline: [{status,count}] }] (SOLO scope global; excluye archived=1; orden lo aplica el frontend)
}
postLite = ALLOWLIST estricta { id, title, publish_date, status, approval_state, platform, content_type, client_id } (jamas notes_team/notes_people/caption: el shape nace minimo, propiedad de seguridad del diseno actual).

SQL AGREGADO (D1/SQLite, todo con bind params, fechas calculadas en el Worker y pasadas como strings YYYY-MM-DD; aprovecha indices existentes idx_mkt_posts_date(client_id, publish_date) e idx_mkt_posts_status(client_id, status)):
Q1 pipeline: SELECT status, COUNT(*) n FROM mkt_posts WHERE client_id=?1 AND publish_date>=?2 AND publish_date<?3 GROUP BY status. Q1b noDate: SELECT COUNT(*) FROM mkt_posts WHERE client_id=?1 AND publish_date IS NULL.
Q2 approvals: SELECT <postLite> FROM mkt_posts WHERE client_id=?1 AND approval_state IN ('pending','changes') AND client_visible=1 ORDER BY COALESCE(publish_date,'9999') ASC LIMIT 5; + COUNT(*) con mismo WHERE (misma definicion de 'pending' que clientCounts, los badges del sidebar dependen de ella).
Q3 semana: SELECT <postLite> FROM mkt_posts WHERE client_id=?1 AND publish_date>=?hoy AND publish_date<?hoyMas7 ORDER BY publish_date ASC, position ASC LIMIT 20.
Q4 atrasados: SELECT <postLite> FROM mkt_posts WHERE client_id=?1 AND publish_date IS NOT NULL AND publish_date<?hoy AND status!='publicado' ORDER BY publish_date ASC LIMIT 10; + COUNT(*) gemelo. (Definicion: fecha pasada y no publicado, exactamente el alcance pedido.)
Q5 donut: SELECT COALESCE(NULLIF(TRIM(platform),''),'Otra') p, COUNT(*) n FROM mkt_posts WHERE client_id=?1 AND publish_date>=?2 AND publish_date<?3 GROUP BY p (re-bucketing final contra PLATTFORMS en JS del Worker).
Q6 racha: SELECT substr(datetime(created_at,'-300 minutes'),1,10) d, COUNT(*) n FROM mkt_activity WHERE created_at>=datetime('now','-35 days') AND (?1 IS NULL OR client_id=?1) GROUP BY d ORDER BY d (created_at es UTC; -300 min = Cancun); streak calculado en el Worker iterando hacia atras desde today.
GLOBAL (sin client_id, NUNCA el patron N+1 de clientCounts): Q7 SELECT client_id, status, COUNT(*) GROUP BY client_id, status (rango mes); Q8 SELECT client_id, COUNT(*) WHERE approval pending/changes AND client_visible=1 GROUP BY client_id; Q9 SELECT client_id, COUNT(*) WHERE publish_date<?hoy AND status!='publicado' GROUP BY client_id; Q10 SELECT client_id, MIN(publish_date) WHERE publish_date>=?hoy GROUP BY client_id; JOIN en memoria del Worker contra la lista de mkt_clients WHERE archived=0; los agregados de counters globales se suman de Q7-Q9 sin queries extra.
INDICES NUEVOS (migracion 004, idempotentes): CREATE INDEX IF NOT EXISTS idx_mkt_posts_publish ON mkt_posts(publish_date) (cubre Q3/Q4/Q9/Q10 globales sin client_id en el prefijo) y CREATE INDEX IF NOT EXISTS idx_mkt_activity_created ON mkt_activity(created_at) (Q6 y de paso corrige el full-scan documentado de GET /activity).
