# Modelo de datos v2 + API v2: migraciones 004/005 (prioridad, tags, checklist, notificaciones, automatizaciones por recetas, vistas guardadas, indices correctivos), endpoints nuevos (PATCH /posts/bulk, POST /posts/:id/duplicate, GET /stats, /notifications, /automations CRUD, /checklist CRUD, /views CRUD, POST /cron) sobre el catch-all existente, mas capa de servicios ES modules en marketing/js/services/ para que las vistas enterprise consuman todo con UI optimista.

## UX
Este modulo define el CONTRATO de interaccion que las vistas consumen. Reglas transversales: toda mutacion es optimista (mutar store, repintar, fetch; si falla revertir + toast en espanol con err.message), errores siempre {error:"mensaje"} via api.js, cero spinners en ediciones de celda, roles verificados server-side en CADA endpoint (isStaff para todo lo nuevo salvo stats reducido y notifications propias). Interacciones concretas: (1) SELECCION MASIVA: el usuario selecciona N posts (checkbox desktop, long-press movil), la barra inferior ofrece Cambiar estado / Mover N dias / Prioridad / Asignar / Visibilidad; al confirmar, services/bulk.js aplica el patch a los N posts del store al instante y manda UN solo PATCH /posts/bulk {ids, patch}; la respuesta trae los posts actualizados con los que el store se reconcilia (resuelve el last-write-wins parcial); si falla, restaura el snapshot completo y toast "No se pudo aplicar el cambio masivo, intenta de nuevo". shift_days permite "Mover 7 dias" para replanificar una semana (caso #1 de agencia). (2) DUPLICAR: menu de tarjeta/fila > "Duplicar"; POST /posts/:id/duplicate devuelve el post nuevo (titulo + " (copia)", approval_state pending, checklist copiada sin marcar); se inserta en el store y se abre su editor; toast "Contenido duplicado". (3) NOTIFICACIONES: la campana del header y el tab Avisos muestran badge con unread; services/notifications.js hace polling de GET /notifications/unread-count cada 60s + en visibilitychange + tras cada mutacion propia, y emite evento 'notif:count' que el shell escucha; abrir el panel hace GET /notifications?filter=..., tabs Todas / No leidas / Menciones / Asignadas; tap en una fila navega al deep-link guardado en la fila (#post=<id>&comment=<id> abre el editor anclado al comentario) y marca leida (POST /notifications/read {ids:[id]}); boton "Marcar todas como leidas" manda {all:true}. Los textos llegan YA resueltos del servidor ("Vianey aprobo Reel Cenotes de Hotel X"): el frontend solo pinta. (4) AUTOMATIZACIONES: NO hay builder generico; pantalla de admin con 8 recetas predefinidas como tarjetas con frase en espanol ("Cuando un post pase a Revision, avisar al cliente") y switch on/off; tocar la tarjeta abre la edicion de 2-3 huecos (selects: horas de recordatorio, dias antes, estado destino); guardar = POST o PATCH /automations; apagar el switch = PATCH {enabled:0} sin perder config. Las recetas de evento disparan dentro del request (tras el UPDATE, en try/catch: jamas rompen la respuesta); las de tiempo corren via POST /cron (GitHub Actions diario + lazy-cron de respaldo colgado de unread-count). Anti-loop: cambios hechos por una automatizacion no re-disparan recetas; recetas diarias deduplican con clave unica (automation_id, post_id, fecha) en mkt_automation_runs: una notificacion por post por dia, nunca spam. (5) CHECKLIST: dentro del editor del post (tab Detalles), lista de subtareas con checkbox; marcar = PATCH /checklist/:id {done:1} optimista (guarda done_by/done_at server-side), agregar = input inline + Enter (POST /posts/:id/checklist, el input queda listo para encadenar), reordenar = POST /posts/:id/checklist/reorder {ids}; el kanban/tabla pueden pedir GET /posts?include=checklist para pintar chip "2/5". Interno: el cliente JAMAS lo recibe. (6) VISTAS GUARDADAS: chips tipo pestanas sobre la vista ("Todo", "Esperando cliente", "Esta semana"); guardar la configuracion actual (filtros+orden+agrupacion+columnas) = POST /views; cambiar de chip aplica config al store sin re-fetch; is_shared=1 la publica al equipo; persistencia confiable en D1 (el dolor documentado de Monday), localStorage solo cachea la ultima vista activa. (7) STATS: el dashboard hace UN solo GET /stats y pinta counters, baterias por cliente, proximos 7 dias y vencidos; cada elemento lleva el filtro equivalente para drill-down a tabla/calendario. Copys de exito: "Guardado.", "Cambio aplicado a N contenidos.", "Receta activada.". Sin em-dashes en ningun texto visible.

## Movil (390px)
Disenado para 390px primero, porque el 99% del uso de IVAE es telefono. (a) Notificaciones: badge numerico en la campana del topbar Y en el tab Avisos de la bottom-nav; el panel es overlay a PANTALLA COMPLETA (no dropdown): header sticky con titulo + "Marcar todas como leidas", tabs como segmented control deslizable, filas de minimo 56px (punto azul no-leida, actor, texto 2 lineas max, tiempo relativo), tap en toda la fila navega y marca leida; menu de 3 puntos de 44px para marcar no-leida. Polling de unread-count (payload de ~30 bytes) en vez de websockets: barato en datos moviles y compatible con vanilla. (b) Bulk: long-press (~500ms) entra en modo seleccion; barra inferior fija con padding-bottom env(safe-area-inset-bottom), botones de 44px (Estado, Mover dias, Prioridad, X); cada accion abre bottom sheet con opciones como filas de color de ancho completo; "Mover N dias" usa stepper +/- grande; el endpoint /posts/bulk garantiza 1 request aunque selecciones 50 posts (clave en red celular). (c) Checklist: filas de 48px, checkbox de 28px con area de tap de 44px, input de alta con teclado abierto y Enter encadenado; el chip "2/5" en tarjetas es solo lectura. (d) Automatizaciones: lista vertical de tarjetas-receta (1 por fila) con switch a la derecha de 44px; editar abre bottom sheet wizard de pasos verticales con selects NATIVOS (cero dropdowns flotantes); es pantalla de admin, poco frecuente, pero funciona completa en telefono (a diferencia de Monday que lo niega en movil). (e) Vistas guardadas: chips en tira horizontal scrolleable (scroll-snap) bajo el view-switcher; chip activo con fondo grad-soft; "+ Guardar vista" al final abre sheet con input de nombre + toggle Compartida. (f) Stats: el contrato devuelve todo en un fetch para que el dashboard movil pinte sin waterfall; upcoming ya viene ordenado y limitado para render directo como lista. (g) Deep-links de notificacion usan location.hash (#post=...) para que el boton atras del telefono funcione y el editor se abra como overlay full-screen. Todos los targets minimo 44px; los servicios re-fetchean en visibilitychange (la duena vuelve a la pestana) en vez de pull-to-refresh custom.
