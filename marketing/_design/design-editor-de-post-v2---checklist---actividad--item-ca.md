# Editor de post v2 + checklist + actividad (item card estilo Monday: bottom sheet movil / modal desktop, 5 tabs, autosave, duplicar, plantillas)

## UX
APERTURA Y CICLO DE VIDA. Toda vista (chip de calendario, card de tablero, fila de tabla, item de agenda) abre el post via editorOpen(postId, {tab}) que hace pushState '#post=<id>' (deep-link compartible: al cargar app.html con #post= el editor se abre tras loadPosts; '#post=<id>:checklist' abre en un tab concreto). Render inmediato de skeleton (header + tabs + filas shimmer) mientras corre GET /posts/:id; en error toast + cierre. Cierre: boton X, Esc en desktop, o boton atras del telefono (popstate). SIEMPRE antes de cerrar se hace flush del autosave; si quedan cambios sin guardar por fallo de red, sheet de confirmacion 'Hay cambios sin guardar' con tres acciones: Reintentar / Descartar cambios / Seguir editando. Nunca se cierra perdiendo texto en silencio.

HEADER (sticky). Fila 1: X (44px), indicador de guardado con 4 estados visibles y aria-live=polite ('Guardado' check gris, 'Guardando…' spinner 12px, 'Sin conexion, se guardara al volver' ambar, 'No se guardo' rojo + boton Reintentar), menu 3 puntos (Duplicar, Copiar enlace, Eliminar en rojo; 'Guardar como plantilla' queda detras de flag v1.5). Fila 2: titulo editable inline (textarea autosize 1-2 lineas, maxlength 200, autosave). Fila 3: chips tocables de acceso rapido: statusBadge (tap abre el picker de estado SIN ir al tab Contenido, es el 80% de los toques), approvalBadge, chip de fecha corta (tap abre date picker nativo). Editar desde chips usa el mismo motor patchField: optimista + rollback + toast.

TABS. Segmented control sticky bajo el header con 5 tabs: Contenido, Guion, Checklist (badge '3/5'), Conversacion (badge con numero de comentarios), Actividad. Cambiar de tab hace flush del autosave y conserva la posicion de scroll por tab. Render perezoso: solo el tab activo monta DOM. El tab inicial por defecto es Contenido; el ultimo tab usado por post se recuerda en sessionStorage.

TAB CONTENIDO. Filas etiqueta-valor de 48px tap-to-edit (patron Monday movil: nunca teclado inline salvo campos de texto): Estado (picker con las 8 etapas como filas de color de ancho completo, tap aplica, cierra y dispara refreshClientCounts), Aprobacion (solo staff, con aviso 'Esto fuerza la decision del cliente'), Fecha (input type=date nativo), Plataforma, Tipo (al cambiar tipo, si la checklist esta vacia ofrece 'Aplicar checklist de <tipo>'), Grabacion (Sin prioridad / G1-G5), Hecho por (texto libre), switch 'Visible para el cliente' (al apagarlo con aprobacion pendiente avisa 'El cliente dejara de verlo'), Inspiracion URL y Video URL (boton Abrir con rel=noopener; valida http/https y rechaza javascript:), Notas del equipo (expand textarea) y una fila por persona de note_labels del cliente (Notas Jairo, Notas Natalia: merge inmutable de notes_people, mismo patron actual). Cada commit de picker = PATCH inmediato de UN campo.

TAB GUION. Bloques HOOK (hint 'Las primeras palabras venden'), BODY y CTA con textareas autosize; Caption con contador en vivo que marca el corte visible de Instagram (~125) y el maximo (2200); Hashtags con contador N/30 y aviso al pasarse; botones 'Copiar caption' (caption + hashtags via copyText) y 'Copiar guion completo' (HOOK/BODY/CTA etiquetados). Autosave con debounce 800ms + flush en blur.

TAB CHECKLIST. Cabecera con barra de progreso (div flex proporcional, verde) + texto '3 de 5 completados'. Cada item: fila 48px con checkbox (zona izquierda completa togglea: target enorme), label (tap edita inline, Enter guarda, Esc cancela), meta sutil cuando esta hecho ('Natalia · hace 2 h' desde done_by/done_at) y menu 3 puntos (Subir, Bajar, Eliminar). Marcar es optimista: barra, badge del tab y contador se actualizan al instante; PATCH detras; rollback + toast si falla. Al llegar a 100%: la barra pulsa verde y, si el status del post va atras en el pipeline, toast accionable 'Checklist completa. Mover a Revision?'. Reorden: drag con pointer events en desktop; en movil flechas Subir/Bajar (sin drag fragil). Add-row sticky al fondo: input 'Agregar paso…', Enter crea (POST) y conserva el foco para encadenar. Vacia: empty state con boton 'Usar plantilla de <tipo>' que inserta los pasos sugeridos del content_type via bulk (ej. Reel: Guion aprobado, Grabar, Editar, Subtitulos y musica, Portada, Caption y hashtags, Enviar a aprobacion, Programar).

TAB CONVERSACION. Hilo con el patron visual existente (avatar, autor, tag 'Interno' ambar con candado, timeAgo); los comentarios del cliente (request-changes) caen aqui. Composer sticky abajo: textarea autosize + toggle de visibilidad con dos estados inequivocos: 'Solo equipo' (default para staff, fondo ambar) vs 'Visible para <nombre del cliente>' (icono de ojo, borde distinto) para que un comentario interno JAMAS se filtre por descuido + boton Comentar (Cmd/Ctrl+Enter en desktop). Envio optimista con re-fetch del hilo en background.

TAB ACTIVIDAD. Timeline read-only agrupado por dia que mezcla mkt_approvals (Aprobado / Cambios pedidos con el comentario del cliente citado) y mkt_activity del post (status.change como 'guion -> grabacion', post.update, post.create, comment.create, post.duplicate, checklist.done). Chips de filtro: Todo / Decisiones / Cambios de estado. 'Cargar mas' en paginas de 40. Responde 'quien movio esto y cuando' sin salir del post.

AUTOSAVE (motor transversal). Mapa dirty por campo. Textos: debounce 800ms tras la ultima tecla + flush en blur, cambio de tab, cierre y visibilitychange hidden (fetch keepalive). Pickers y switches: guardado instantaneo. El PATCH manda SOLO los campos dirty (se elimina el PATCH de payload completo del drawer viejo, que pisaba todo). Exito: merge al snapshot + upsertPost en el store + evento 'post:updated' (las vistas re-renderizan quirurgico). Fallo: el dirty se conserva, estado 'No se guardo' con Reintentar, lo tecleado nunca se pierde. window offline/online: pasa a 'Sin conexion' y reintenta solo al volver. Si el backend implementa expected_updated_at y responde 409: banner 'Alguien mas edito este post' con boton 'Actualizar' que recarga el snapshot y re-aplica los campos dirty no conflictivos.

DUPLICAR. Menu 3 puntos -> sheet con switches: Incluir checklist (se copia con done en 0), Incluir guion y caption; la fecha SIEMPRE nace vacia (no choca el calendario). POST /posts/:id/duplicate -> toast 'Contenido duplicado' con accion 'Abrir' que abre el editor del nuevo. El duplicado nace con titulo + ' (copia)', status idea, approval_state pending; no copia comentarios ni aprobaciones.

PLANTILLAS DE POST. El boton 'Nuevo contenido' (todas las vistas) abre un sheet selector: 'Vacio' + plantillas builtin por tipo (Reel, TikTok, Carrusel, Historia, Foto, Pauta) mostrando que incluye cada una (estructura de guion con placeholders, checklist sugerida, plataforma default). Elegir una hace POST /posts prefilled + POST checklist/bulk y abre el editor con foco en el titulo. Las builtin viven en templates.js (cero red, cero migracion); plantillas custom por cliente (mkt_templates) son extension v1.5 ya prevista en el esquema.

ELIMINAR. openConfirm danger existente -> DELETE /posts/:id -> cierra editor, evento 'post:deleted', toast. Sin undo (el delete es hard en el backend actual); el copy lo deja claro.

ACCESIBILIDAD. role=dialog aria-modal, foco atrapado y retorno de foco al disparador; tablist/tab/aria-selected en los tabs; indicador de guardado aria-live; targets de 44px; :focus-visible con --ring existente; prefers-reduced-motion anula las transiciones del sheet.

DESKTOP (>=720px). Mismo componente como modal centrado max-width 760px, max-height 90dvh, radios completos; los pickers son popovers anclados (reusa openPopover de app.js con su flip/clamp/teclado) en vez de bottom sheets; Esc cierra con flush; drag de checklist con pointer; el resto del flujo es identico, un solo codigo con dos contenedores.

## Movil (390px)
DISEÑADO PRIMERO A 390px. El editor es un sheet a PANTALLA COMPLETA (100dvh, nunca 100vh por la barra de URL) que entra con translateY 250ms ease-out. Sin backdrop-dismiss y sin swipe-down para cerrar: es captura de datos (regla NN/g); se cierra solo con X o con el boton atras del telefono (popstate). Layout vertical: header sticky 56px (X 44px a la izquierda, indicador de guardado a la derecha, menu 3 puntos 44px), bloque de titulo + chips de acceso rapido (status/aprobacion/fecha editables con 2 taps sin navegar tabs), tabs sticky de 44px con scroll horizontal, scroll-snap y mascara de fade en los bordes (el badge 3/5 del checklist visible sin scrollear), y area de contenido con overflow-y auto, overscroll-behavior: contain (sin scroll chaining al body) y padding-bottom calc(72px + env(safe-area-inset-bottom)).

PICKERS SECUNDARIOS: bottom sheets parciales (max-height 70dvh, asa de 40x4, animacion 200ms) apilados SOBRE el editor, cada uno con su propio pushState: el boton atras cierra el picker activo, no el editor (stack de history gestionado por el componente sheet compartido). Estos SI cierran con tap en backdrop porque son menus, no formularios. Opciones de picker como filas de color de ancho completo, minimo 48px: imposible fallar con el pulgar.

TECLADO: composer de comentarios y add-row de checklist sticky bottom; listener de visualViewport.resize reposiciona la barra para que el boton Comentar no quede tapado por el teclado de iOS; todos los inputs con font-size 16px (evita el zoom automatico de iOS); inputmode y type correctos (date nativo para fecha, url para links). El textarea del titulo y los del guion crecen con autosize sin saltos de layout.

TARGETS Y GESTOS: todo elemento tocable >= 44px. En la checklist hay dos zonas de tap claras por fila: mitad izquierda (checkbox extendido) togglea, label edita. Reorden movil con flechas Subir/Bajar en el menu del item, no drag (el long-press drag en 390px es fragil y la lista es corta). Sin swipe actions en v1: toda accion vive en menus visibles.

RENDIMIENTO: render perezoso por tab (solo el activo monta DOM, los demas al primer toque); el GET /posts/:id trae post + comments + approvals + checklist en UNA respuesta (cero waterfall); actividad se pide solo al abrir su tab. Re-render quirurgico por seccion (barra de progreso, badge, fila), nunca replaceChildren del sheet completo mientras se teclea. Animaciones 150-250ms; prefers-reduced-motion las anula. Autosave + fetch keepalive en pagehide cubren el caso real movil: la dueña edita, bloquea el telefono y espera que todo este guardado.

## Contrato de datos
STORE COMPARTIDO (lee): state.me {id,name,role}, state.activeClientId, state.posts[] (para sync optimista de vistas), activeClient().note_labels (filas de notas por persona). STORE (escribe): upsertPost(post) tras cada PATCH ok; removePost(id) tras DELETE; emite eventos 'post:updated' {id, fields}, 'post:created' {post} (duplicar/plantilla), 'post:deleted' {id}; las vistas se suscriben y re-renderizan quirurgico; tras cambios de status/approval_state/client_visible invoca el hook refreshClientCounts() existente.

ESTADO LOCAL DEL EDITOR (no vive en el store): {postId, snapshot (post del servidor), dirty: Map<campo,valor>, saveState: 'saved'|'dirty'|'saving'|'error'|'offline', lastSavedAt, tab, scrollPorTab, checklist: [{id,label,done,position,done_by,done_at}], comments[], approvals[], activity[], pickerStackDepth}.

API QUE CONSUME. LECTURA: GET /posts/:id -> {post, comments, approvals, checklist:[]} (respuesta existente EXTENDIDA con checklist para staff; para rol client la respuesta no cambia: checklist y activity son internos y publicPost es allowlist); GET /activity?post_id=<id>&limit=40 (tab Actividad, lazy). ESCRITURA: PATCH /posts/:id con SOLO los campos dirty (endpoint existente, acepta parcial; opcional expected_updated_at -> 409 {post}); POST /posts/:id/checklist {label} -> 201 item; PATCH /posts/:id/checklist/:itemId {label?, done?} (el server setea done_by/done_at); DELETE /posts/:id/checklist/:itemId; POST /posts/:id/checklist/reorder {updates:[{id,position}]} (env.DB.batch, espejo de posts/reorder); POST /posts/:id/checklist/bulk {items:[{label,position}]} (plantillas); POST /posts/:id/duplicate {include_checklist:0|1, include_script:0|1} -> 201 {post}; POST /posts/:id/comments {body, internal:0|1} (existente); POST /posts (existente, prefilled por plantilla); DELETE /posts/:id (existente). Todo via api.js (cookie mkt_session same-origin, timeout 30s, errores en espanol); cero campos nuevos enviados que no esten en POST_EDITABLE_FIELDS.

D1 (migracion, 100% idempotente, SIN ALTERs): CREATE TABLE IF NOT EXISTS mkt_checklist_items(id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))), post_id TEXT NOT NULL REFERENCES mkt_posts(id) ON DELETE CASCADE, label TEXT NOT NULL, done INTEGER NOT NULL DEFAULT 0, position INTEGER NOT NULL DEFAULT 0, done_by TEXT REFERENCES mkt_users(id) ON DELETE SET NULL, done_at TEXT, created_at TEXT NOT NULL DEFAULT (datetime('now'))); CREATE INDEX IF NOT EXISTS idx_mkt_checklist_post ON mkt_checklist_items(post_id, position); CREATE TABLE IF NOT EXISTS mkt_templates(id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))), client_id TEXT REFERENCES mkt_clients(id) ON DELETE CASCADE, name TEXT NOT NULL, content_type TEXT NOT NULL DEFAULT 'reel', config TEXT NOT NULL DEFAULT '{}', created_by TEXT REFERENCES mkt_users(id) ON DELETE SET NULL, created_at TEXT NOT NULL DEFAULT (datetime('now'))); CREATE INDEX IF NOT EXISTS idx_mkt_templates_client ON mkt_templates(client_id). REGLA DE SEGURIDAD: checklist y plantillas son SIEMPRE internas; publicPost (allowlist POST_RETURN_FIELDS) no se toca, el portal de cliente no recibe ni un byte nuevo. Todos los handlers nuevos re-verifican post.client_id contra el scope (patron approve/comments existente) y gatean con isStaff.

## API
- GET /posts/:id (extender handler existente): incluir checklist:[] ordenada por position en la respuesta para staff; respuesta para rol client SIN cambios (publicPost allowlist intacta, checklist jamas viaja al portal)
- PATCH /posts/:id parcial (ya existe); OPCIONAL pero recomendado: aceptar expected_updated_at y responder 409 {post} si difiere, para que el autosave detecte ediciones concurrentes
- POST /posts/:id/checklist {label} -> 201 item (isStaff + re-check de post.client_id, patron de approve/comments)
- PATCH /posts/:id/checklist/:itemId {label?, done?}: el server setea done_by=session.user_id y done_at al marcar, los limpia al desmarcar
- DELETE /posts/:id/checklist/:itemId
- POST /posts/:id/checklist/reorder {updates:[{id,position}]} aplicado con env.DB.batch (espejo de POST /posts/reorder, validando pertenencia al post)
- POST /posts/:id/checklist/bulk {items:[{label,position}]} para insertar plantillas de checklist en una llamada
- POST /posts/:id/duplicate {include_checklist:0|1, include_script:0|1} -> 201 {post}: copia campos editables, title + ' (copia)', status 'idea', approval_state 'pending', publish_date NULL, position al final, copia notes_people; copia checklist con done=0 si se pide; NO copia comments ni approvals; logActivity 'post.duplicate'
- GET /activity?post_id=<id>&limit=40: agregar filtro post_id al handler de activity existente (mas CREATE INDEX IF NOT EXISTS sobre mkt_activity(post_id, created_at) en la migracion si se quiere barato)
- logActivity: nuevos verbos 'post.duplicate', 'checklist.add', 'checklist.done', 'checklist.delete' (best-effort, mismo helper) + entradas en ACTION_LABELS del frontend
- OPCIONAL v1.5: GET/POST/DELETE /templates (CRUD de mkt_templates scoped por cliente); v1 funciona solo con plantillas builtin en JS sin tocar red
